程序员系列（1）—— 程序员，你真的会写代码吗？

可读性基本定理

Code should be written to minimize the time it would take for someone else to understand it.

代码的写法应当使别人理解它所需的时间最小化。

引子

写代码是程序员的基本功。但是，程序员，你真的会写代码吗？今天，我们不谈什么高大上的“架构”，就谈谈怎么写代码。

作为程序员，你看过、写过、改过、删过的代码可能不计其数。然而，回头想想，真正让你感到赏心悦目、拍手称快的次数又有多少呢？要知道，你读代码的时间远远大于你实际写代码的时间。如果代码可读性不好，你的工作效率肯定会大打折扣。特别是，对于一个团队来说，如果没有良好的、统一的代码规范，协同工作效率必然很受影响，造成不必要的时间浪费。

《Clean Code》的作者Bob大叔认为“在代码阅读过程中人们说脏话的频率是衡量代码质量的唯一标准。” 哈哈。太TM经典了。

可读性为什么重要

书中观点

软件成本 = 开发成本 + 维护成本。往往，维护成本>>开发成本。这其中耗费的主要成本就是由于理解代码和修改代码造成的。

——《Structured Design》

个人观点

可读性永远是第一位的。代码的可读性与效率、架构、易于测试一点也不冲突。
代码可读性好、bug少是对程序员的基本要求。

就我亲身经历而言。我”见识“过很多奇葩的代码。主要包括以下几种情形。

  1. 职责划分不清

    • 一个方法超过一百行，一个类一两千行
    • 各种代码杂糅在一起，该抽取的不抽取，可复用性很低
    • 不使用MVC之类的模式，代码不分层，甚至所有代码逻辑都写在展示层

  2. 命名乱七八糟

    • 有的用英语单词，有的用汉语拼音
    • 有的方法明明是做更新，名称却是getXXX()
    • 缩写不规范，容易造成误解
    • 团队中没有统一的编码规范，每个人风格各异

前言

本书旨在帮助你把代码写得更好 。。。我们希望本书对你每天的编程工作有很多帮助，并且希望你把本书推荐给你团队中的每个人。

各章总结

表面层次的改进

  1. 一定要让命名可以“自解释”，尽量具体形象。不要用汉语拼音，也不要随便缩写，可读性比变量长短重要的多
  2. 不要使用有二义性的词
  3. 注意代码的美观：分段、锁紧、空行、适当的对齐、顺序的一致性
  4. 好代码>坏代码+好注释。为缺陷加TODO注释。为特殊的地方写注释
  5. 使用明确、恰当、含义丰富的词精确描述函数的行为。注释中可以给出I/O的例子

    简化循环和逻辑

  6. 形式上嵌套一般不超过2层，能return的尽早return。这个原则可以让你大大降低if-else的嵌套深度
